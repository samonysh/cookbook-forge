# O'Reilly CookBook 行文布局参考

> 本文件是 Cookbook Generator 生成章节时的写作风格参考。写作时不必照搬文字，但必须遵循其结构与节奏。

## O'Reilly CookBook 系列核心特征

### 1. 问题驱动（Problem / Solution / Discussion）

O'Reilly CookBook 的灵魂是 Recipe 三段式：

- **Problem**：用一句话精准描述问题场景（"如何在 Flask 中实现基于 Token 的认证？"）。
- **Solution**：给出最短可运行代码或命令，让读者立刻能跑通。
- **Discussion**：解释为什么这样做、关键参数含义、边界条件、替代方案、性能/安全权衡、常见错误。

写 Recipe 章节时，每个小节都应当能套进这个模板。即使是概念章，也要把"为什么需要 X"作为第一句。

### 2. 章节开头的"本章速览"

每章开头 1–2 页：

- 这章要解决什么问题域。
- 读者读完后能做什么。
- 这章在全书中的位置（前置知识 / 后续章节）。
- 小节列表 + 一句话说明。

### 3. 代码先于文字

O'Reilly 的排版节奏是：**给出代码 → 解释代码 → 给出变式**。不要先讲三段理论再给代码。读者是来解决问题的，先看到能跑的东西会更有耐心读后面的解释。

代码要求：

- 必须是真实可运行的（不要伪代码，除非明确标注）。
- 每行/每个关键步骤有注释。
- 文件路径、版本号、依赖明确标出。
- 长代码拆成多段，每段前后都有解释。

### 4. 图表用于"建立心智模型"

O'Reilly 的图不多，但每张图都回答一个具体问题：

- 架构图：组件之间谁调用谁？
- 时序图：一次请求经过哪些环节？
- 状态机图：对象的生命周期是什么？
- 流程图：决策点在哪里？

每张图必须有编号和图注，正文中必须引用（"如图 4-2 所示…"）。不要放装饰性的图。

### 5. "See Also" 交叉引用网

每章/每个 Recipe 末尾的"See Also"指向：

- 本书其他章节（鼓励读者跳转）。
- 官方文档、RFC、论文、经典博客。

这让书变成一个知识网络，而不是线性阅读。

### 6. "Warning / Note / Tip" 三类标注

- **Warning**（红色/警告）：会导致数据丢失、安全漏洞、生产故障的坑。
- **Note**（蓝色/提示）：容易忽略但重要的细节。
- **Tip**（绿色/技巧）：能让代码更优雅或性能更好的小窍门。

本 Cookbook Generator 在 MDX 中统一用 callout 卡片表达：

- `callout pitfall` → Warning
- `callout keypoints` → Note
- `callout recipe` → Tip/Recipe

### 7. 面向实践者的"深度"

O'Reilly CookBook 不是入门书，也不是纯理论书。深度体现在：

- 不只讲 API 怎么调用，还讲内部实现、设计权衡。
- 不只讲 happy path，还讲错误处理、并发、性能、调试。
- 不只有"怎么做"，还有"为什么不那样做"。

写原理部分时，至少要回答一次"为什么这样设计"。

## Springer Handbook 系列核心特征（补充用于综述/手册类章）

### 1. 分层结构 + 全景图先行

Springer Handbook 开篇必有一张"全书地图"（Taxonomy / Roadmap），把领域分成若干子领域，每章对应一个节点。综述章节也遵循"先总后分"：

- 第一节：领域定义、历史、分类法。
- 后续：按分类法每类一章/一节。
- 末尾：对比表（不同方法的优劣、适用场景）。

### 2. 术语表 + 符号表

Handbook 会在章首或附录列出本章用到的：

- 关键术语（中英对照 + 定义）。
- 数学符号（如果有公式）。
- 缩写表。

本生成器在附录中自动生成术语表。

### 3. 对比表格密度高

Springer Handbook 大量使用表格做横向对比：

- 不同算法的时间/空间复杂度。
- 不同工具的特性矩阵。
- 不同研究流派的前提/结论对比。

写综述/对比章节时，至少有 1–2 张对比表。

### 4. 引用规范

每个关键论断都有引用。本生成器统一在章末列出参考文献，并在 `metadata/sources.md` 维护完整来源。

## 写作节奏建议（每章 4000 字左右）

```text
[100 字] 章节引入：这章解决什么问题 + 与全景图的关系
[200 字] 本章地图 callout
[每节 800–1200 字]
  - 原理引入 (200 字)
  - 代码/示例 (带注释)
  - 表格 (如有)
  - 图 (如有) + 解释
  - 关键讨论 (300 字)
[600 字] Recipes (2–3 个)
[300 字] 常见陷阱 callout
[200 字] 要点总结 callout
[200 字] 延伸阅读 + 参考文献
```

## 风格禁忌

- ❌ 教材式："本章我们将学习…"
- ❌ 营销式："XX 是世界上最好的…"
- ❌ 翻译腔：长句堆叠、被动语态泛滥
- ❌ 不写注释的大段代码
- ❌ 没有图的架构章 / 没有代码的实操章
- ❌ 空话："XX 非常重要""XX 性能很高"（要给数据/基准/解释为什么）

## 风格鼓励

- ✅ 第二人称对话式："你会发现…""你需要…"
- ✅ 短句 + 主动语态
- ✅ 具体数字：版本号、性能数字、时间复杂度
- ✅ "在生产环境中""在 10k QPS 下" 这类场景限定
- ✅ 真实代码 > 伪代码，标注来源文件
- ✅ 跨章链接形成知识网络
